# 未来开发方向与扩展分析

> 本文档从信安赛作品赛的评审角度出发，分析项目在功能完整性、新功能开发、密码学技术应用和安全性四个维度的扩展方向。

---

## 一、功能完整性补全（当前实现距离生产级仍存在的缺口）

### 1.1 持久化数据库

**现状**：开发环境使用内存 + JSON 文件存储。重启后数据丢失风险高，JSON 文件在并发写入时存在数据损坏风险。

**扩展方向**：
- 引入 PostgreSQL/MySQL 作为主存储
- 设计数据库 schema：users 表、plans 表、shares 表、inheritance_requests 表、audit_logs 表
- 使用 TypeORM 或 Prisma 作为 ORM 框架
- 添加数据库迁移脚本管理

**优先级**：高（资产类应用必须有可靠的持久化存储）

### 1.2 用户系统完善

**现状**：
- 用户注册字段仅包含 name、email、password
- 登录使用验证码 + 密码，缺少多因素认证
- 无邮箱验证流程
- 无密码找回功能
- 无管理员角色

**扩展方向**：
- 实现完整的邮箱验证流程（注册验证链接）
- 添加 TOTP/WebAuthn 多因素认证
- 实现基于角色的访问控制（RBAC）：管理员、普通用户、只读审计员
- 添加登录审计日志（IP、设备、时间）
- 实现会话管理和安全登出机制

**优先级**：高

### 1.3 份额刷新后的监护人侧前端支持

**现状**：
- server 端已完成同态份额刷新的密码学实现
- 监护人通过邮件接收刷新增量
- 但前端没有任何"本地更新份额"的功能页面
- 监护人需要手动修改本地记录的份额值，体验差且易出错

**扩展方向**：
- 设计份额保管箱页面：展示监护人名下所有份额及最新更新时间
- 增量应用一键确认：前端计算 v'_i = v_i + delta_v(i) 并更新本地显示
- 份额导出功能：加密导出份额备份（QR 码/加密文件）
- 份额导入功能：在新设备上恢复已持有的份额

**优先级**：高（直接影响同态刷新功能的可用性）

### 1.4 继承完成后的资产展示

**现状**：
- 资产恢复后仅通过邮件向继承人发送
- 系统中没有安全的在线资产查看页面
- 文件资产在邮件中以附件形式发送，大文件受限

**扩展方向**：
- 设计安全的"遗产接收"页面，继承人在身份验证后在线查看资产
- 实现到期自动删除机制（资产查看 30 天后自动销毁）
- 支持大文件通过分块下载或网盘链接方式交付
- 添加数字遗产证书（PDF 格式的正式继承报告）

**优先级**：中

### 1.5 单元测试与集成测试

**现状**：
- 项目 test 脚本引用了 Jest，但未编写任何测试用例
- 密码学核心（shamir.ts）缺少正确性验证测试
- 缺少集成测试覆盖创建→分发→恢复的完整流程

**扩展方向**：
- 密码学单元测试：Shamir 拆分与恢复的 100 轮随机正确性验证
- Pedersen 承诺的同态验证测试
- 零和多项式刷新增量的正确性测试（刷新后份额仍能恢复同一主密钥）
- 胁迫码检测测试
- 端到端集成测试：创建计划→登录→发起继承→提交份额→恢复资产

**优先级**：高（密码学项目必须有测试保障正确性）

### 1.6 前端输入验证增强

**现状**：
- 后端 CreatePlan 有基本的输入验证
- 前端部分表单缺少即时校验
- 无防止 XSS/CSRF 的安全头配置

**扩展方向**：
- 实现前后端双重输入校验
- 添加 Helmet 中间件配置安全 HTTP 头
- 实现 CSRF token 机制
- 使用 DOMPurify 对用户输入进行清理

**优先级**：中

## 二、新功能开发

### 2.1 分布式密钥生成（DKG）

**原理**：无需单一可信创建者，t 位监护人通过交互式协议共同生成主密钥。每位监护人各持一份私钥碎片，无人知道完整私钥。

**优势**：
- 消除"创建者知道完整密钥"的信任假设
- 密钥从诞生一刻起即分布式存在
- 即使创建者本人也无法单方面获取密钥

**参考协议**：Feldman DKG、Joint-Feldman DKG

**开发复杂度**：高（需要 t 轮网络通信 + 零知识证明验证）

**优先级**：中（对安全竞赛作品的密码学深度有显著加分）

### 2.2 门限签名（区块链资产直接转移）

**原理**：EVM 链上部署门限签名合约，t 位监护人提交签名碎片后由合约自动聚合为完整签名并执行转账。

**现状**：
- contracts/ 目录下已有 LegacyContract.sol 的框架
- 未实现真正的门限签名聚合逻辑

**扩展方向**：
- 实现 ECDSA 门限签名的两方或多方协议（基于 GG18/GG20 协议）
- 部署 EVM 合约：接收 t 个签名碎片，验证后执行资产转移
- 资产恢复时跳过"解密→发送"环节，直接链上转账
- 支持多链（Ethereum、Polygon、Arbitrum）

**优先级**：高（与本项目的数字资产继承核心目标直接吻合）

### 2.3 多继承人资产分割

**现状**：
- 当前一份遗产计划仅支持一个继承人
- 无法按比例分割资产给多个继承人

**扩展方向**：
- 支持在创建计划时指定多个继承人和各自的分配比例
- 继承触发后，按比例自动分割资产（对于同质化资产直接分配，对于非质化资产指定归属）
- 实现分割后的独立加密封装（每个继承人获得独立的加密包）

**优先级**：中（扩展应用场景的关键功能）

### 2.4 可编程继承触发器

**现状**：
- 仅支持时间锁和共识两种触发模式
- 触发条件固定不可扩展

**扩展方向**：
- 链上活动检测触发器：指定时间范围内无链上交易自动触发
- 心跳检测触发器：用户需定期"签到"，超时视为失能
- "死亡证明"Oracle 触发器：对接第三方公证服务 API
- 复合触发器：AND/OR 逻辑组合多个条件
- 延迟执行触发器：触发后设 N 天冷却期，期内可取消

**优先级**：中

### 2.5 监护人加密通信通道

**现状**：
- 监护人之间无法直接通信
- 胁迫警报后监护人需自行联系，缺乏正式通道

**扩展方向**：
- 基于 ECIES 的监护人端到端加密聊天
- 计划内公告板（创建者可发布消息，所有监护人可见）
- 继承触发后协商投票机制（确认/质疑继承请求）
- 胁迫恢复流程：警报后监护人协商是否强行推进继承

**优先级**：低

### 2.6 审计追踪与透明性

**扩展方向**：
- 所有关键操作（创建、提交份额、刷新、继承）上链存证
- 生成不可篡改的审计日志（Merkle Tree 哈希链）
- 监护人可独立验证所有操作的时间戳和合法性
- 引入第三方审计接口

**优先级**：低

## 三、新密码学技术应用

### 3.1 零知识证明（ZK-SNARKs/STARKs）

**应用场景**：替代 Pedersen 承诺验证中的 Lagrange 系数查询——监护人证明"我知道一个有效的份额值"而不泄露该值，且不需要公开所有承诺供验证。

**优势**：
- 安全性可降至计算安全假设（无需可信第三方参数）
- 验证成本从 O(t^2) 的 EC 点运算降至常数级
- 支持更大规模的门限（n=100+）
- STARK 无需可信设置，后量子安全

**推荐方案**：
- 短期：circom + snarkjs 实现 zk-SNARK（以太坊生态兼容）
- 长期：STARK（StarkWare 生态，后量子安全）

**优先级**：中（对作品密码学深度有显著提升，但开发量较大）

### 3.2 前向安全性（Forward Security）

**应用场景**：当前同态份额刷新使用零和多项式，所有历史刷新产生的增量文件如果都被攻击者获取，理论上可能累积出主密钥信息。

**解决方案**：引入前向安全性——每次份额刷新后，旧的多项式系数和旧的份额信息被密码学承诺"封装"起来，即使攻击者后续获得了 server 的全部历史增量，也无法逆向推导当前有效的份额。

**优先级**：高（安全性增强、理论基础扎实、适合竞赛展示）

### 3.3 BLS 签名聚合

**应用场景**：在门限签名恢复时，BLS 签名可以在一轮通信中完成签名聚合，而 ECDSA 门限签名需要多轮交互。

**优势**：
- 签名长度短（单个群元素）
- 支持签名聚合（n 个签名压缩为 1 个）
- 无需多轮交互

**所需基础设施**：需支持 BLS12-381 曲线的链（如 Ethereum 2.0 存款合约）

**优先级**：低（依赖链支持，当前阶段性价比不高）

### 3.4 身份基加密（IBE）

**应用场景**：监护人份额邮件加密不需要预先生成和分发 ECIES 密钥对，直接用监护人的邮箱地址作为公钥加密。

**优势**：
- 简化密钥管理流程（无需维护监护人的公钥目录）
- 监护人只需要持有一个全局的主密钥私钥即可解密
- 支持"事后加密"——即使监护人尚未注册系统，创建者仍可用其邮箱地址加密份额

**推荐方案**：基于双线性对的 IBE（Boneh-Franklin 方案）

**优先级**：中

### 3.5 属性基加密（ABE）

**应用场景**：资产可以按属性标记（如"仅 t 名监护人可解密"、"仅继承人可查看"、"30 天后过期"），用 ABE 策略直接控制解密权限。

**优势**：
- 细粒度访问控制内置于密码学
- 策略灵活（支持 AND/OR/NOT 逻辑组合）
- 简化业务逻辑（解密条件由密码学保障，而非代码）

**优先级**：低（理论价值高但实际性能瓶颈明显）

### 3.6 抗量子密码学

**应用场景**：当前使用的 secp256k1 椭圆曲线可被 Shor 算法在量子计算机上高效破解。长远来看，需要迁移到抗量子签名方案。

**推荐方案**：
- CRYSTALS-Dilithium（格基签名，NIST 已标准化）
- CRYSTALS-Kyber（格基密钥封装机制）
- 作为"混合模式"（当前 ECC + 抗量子并行，兼容过度时期）

**优先级**：低（长期方向，短期不影响竞赛评分）

## 四、安全性增强

### 4.1 形式化验证

**方法**：使用定理证明器（如 Coq、Isabelle/HOL）或模型检测器（如 TLA+、ProVerif）对密码学协议进行形式化证明。

**可验证的内容**：
- Shamir 秘密共享的正确性：任意 t 个份额可恢复主密钥
- Pedersen 承诺的绑定性和隐藏性
- 零和多项式 Z(0) = 0 的数学保证
- 同态刷新前后主承诺不变
- 胁迫码的计算不可区分性

**优先级**：中（对学术型竞赛作品有很好的加分效果）

### 4.2 抗侧信道攻击

**现状**：
- 当前 BigInt 运算使用 JavaScript 的 number 类型
- 比较操作可能泄露时间信息
- 未考虑幂侧信道攻击（如通过功耗分析推断密钥位）

**改进方向**：
- 使用 constant-time 比较（字符串比较替换为异或比较）
- 所有密码学运算使用恒定时间算法处理
- 敏感运算中加入随机延迟（混淆时序）
- mock 级攻击测试验证防护效果

**优先级**：低（Web 应用侧信道攻击面有限，但展示安全工程全面性时有加分）

### 4.3 拒绝服务攻击防护

**现状**：
- 无速率限制
- 无请求频率控制
- 无 DDoS 防护

**扩展方向**：
- 全端点添加 Rate Limiting（express-rate-limit）
- 登录失败次数限制和账户临时锁定
- 继承请求的防滥用机制（同一计划 N 小时内限发起一次）
- API 请求的白名单 IP 控制

**优先级**：中

### 4.4 密钥管理强化

**现状**：
- 主密钥随机性依赖 CryptoJS 的 WordArray.random（需要确认强度）
- 无硬件安全模块（HSM）支持

**扩展方向**：
- 使用 crypto.getRandomValues()（Web Crypto API）替代第三方库的随机数生成，确保密码学安全熵源
- 在内存中使用后及时清零敏感变量（对于 BigInt 类型需覆盖置零）

**优先级**：高

### 4.5 安全审计自动化

**扩展方向**：
- 集成 ESLint 安全规则（eslint-plugin-security）
- 添加 npm audit 到 CI/CD 流程
- 集成 Snyk/Dependabot 依赖漏洞扫描
- 编写渗透测试用例（OWASP Top 10 覆盖验证）

**优先级**：中

### 4.6 备份与灾难恢复

**现状**：
- 当前无备份机制
- server 崩溃后数据完全丢失

**扩展方向**：
- 实现自动定期备份到加密存储
- 设计灾难恢复流程：从加密备份恢复数据，通过监护人重新分发份额重建系统
- 多地域冗余部署方案

**优先级**：中

## 六、当前项目优势总结（便于在评审中说明）

1. **密码学创新**：零和多项式同态份额刷新是相比于传统 Shamir 方案的核心创新点，server 零信任模型具备清晰的安全边界。

2. **完整闭环**：覆盖"创建→分发→验证→刷新→恢复"的全生命周期，而非孤立的功能模块。

3. **抗胁迫设计**：胁迫码机制在数字资产继承领域属首创设计，具有明确的安全价值。

4. **开源全栈**：MIT 许可证，前后端全栈开源，技术栈完整且可演示。
